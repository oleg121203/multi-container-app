# ATLAS — Instruction Template for Agent Coding

## Загальна інформація

Цей документ представляє собою детальну інструкцію для розробки багатоагентної системи ATLAS з чіткими фазами реалізації. Система складається з трьох LLM-агентів з розділеними ролями, векторної бази даних для пам'яті, інфраструктури на Kubernetes, та інтеграції з різноманітними інструментами через MCP Hub.

## Фаза 1: Підготовка інфраструктури

### Цілі фази
- Підготувати Kubernetes кластер з необхідними компонентами
- Сконвертувати існуючі docker-compose конфігурації в Kubernetes маніфести
- Налаштувати моніторинг та спостережуваність

### Кроки реалізації
1. **Базова інфраструктура Kubernetes**
   - Створити окремі простори імен (namespaces): core, data, observability, security
   - Налаштувати базові квоти ресурсів та ліміти

2. **Міграція з Docker Compose на Kubernetes**
   - Використати Kompose для початкової конвертації
   - Вручну доопрацювати маніфести, особливо для stateful-сервісів
   - Додати readiness/liveness probes, ресурси, тощо

3. **Налаштування сховищ даних**
   - Підготувати StorageClass і PVC для Qdrant/Milvus
   - Налаштувати Redis з персистентністю
   - Забезпечити резервне копіювання на рівні томів

4. **Встановлення системи моніторингу**
   - Розгорнути Prometheus + Grafana в namespace observability
   - Налаштувати базовий експорт метрик
   - Створити початкові дашборди для спостереження за системою

### Тести і валідація
- **Інтеграційні тести інфраструктури:**
  - Перевірка коректності розгортання всіх компонентів
  - Валідація підключення між сервісами
  - Перевірка персистентності даних після перезапуску подів

- **Тести моніторингу:**
  - Перевірка збору базових метрик системи
  - Валідація дашбордів Grafana
  - Тестування оповіщень про критичні події

### Критерії приймання
- [ ] Успішно розгорнуто Kubernetes кластер з усіма необхідними просторами імен
- [ ] Сконвертовані та доопрацьовані маніфести для всіх сервісів
- [ ] Працюють Prometheus і Grafana з базовими дашбордами
- [ ] Налаштовано StorageClass і PVC для постійного зберігання даних
- [ ] Базові CI/CD конвеєри для розгортання інфраструктури

## Фаза 2: Реалізація основних агентів і пам'яті

### Цілі фази
- Імплементувати LLM1 з інтеграцією RAG
- Налаштувати LLM2 на базі Ollama з інструментами
- Забезпечити базову інтеграцію з Linear

### Кроки реалізації
1. **Впровадження RAG (Retrieval-Augmented Generation)**
   - Налаштувати Qdrant/Milvus для векторних ембедінгів
   - Реалізувати шар індексації та пошуку
   - Налаштувати Redis для короткострокової пам'яті

2. **Реалізація LLM1 (інтерфейс користувача)**
   - Імплементувати абстракцію провайдерів LLM (OpenAI/Mistral/Gemini/Ollama)
   - Налаштувати фолбек-ланцюг між провайдерами
   - Інтегрувати з шаром RAG для контекстних "спогадів"

3. **Реалізація LLM2 (оркестратор)**
   - Налаштувати локальний Ollama з моделлю gpt-oss:latest
   - Інтегрувати AutoGen/MetaGPT для когнітивної оркестрації
   - Реалізувати інструмент для роботи з Linear API

4. **Налаштування базових робочих потоків**
   - Імплементувати потік "користувач просить задачу → issue в Linear"
   - Забезпечити відслідковування статусу задач

### Тести і валідація
- **Тести RAG:**
  - Unit тести для індексації та пошуку в векторній БД
  - Перевірка релевантності знайдених "спогадів"
  - Тестування персистентності даних

- **Тести LLM1:**
  - Перевірка коректності відповідей з урахуванням контексту
  - Тестування фолбек-механізму між провайдерами
  - Навантажувальне тестування для оцінки швидкодії

- **Тести LLM2 і інтеграцій:**
  - Валідація роботи з локальною моделлю Ollama
  - Перевірка коректної роботи з Linear API
  - E2E тести для повного потоку від запиту до створення задачі

### Критерії приймання
- [ ] LLM1 успішно використовує релевантні "спогади" з векторної БД
- [ ] LLM2 працює на локальній моделі Ollama з автоматичним фолбеком при потребі
- [ ] Демонстраційний потік: "користувач просить задачу → issue в Linear" працює коректно
- [ ] Всі основні тести проходять успішно
- [ ] Моніторинг агентів показує базові метрики (латентність, успішність, etc.)

## Фаза 3: Автоматизація та безпека

### Цілі фази
- Впровадити LLM3 для нагляду і безпеки
- Реалізувати MCP Hub для інструментальної інтеграції
- Налаштувати безпечне виконання автоматизації GUI

### Кроки реалізації
1. **Імплементація LLM3 (наглядач/безпека)**
   - Налаштувати інтеграцію з Falco для моніторингу подій безпеки
   - Імплементувати класифікацію і пріоритезацію подій
   - Реалізувати автоматизовані реакції через K8s API

2. **Розробка MCP Hub**
   - Реалізувати базову інфраструктуру MCP (Model Context Protocol) серверів
   - Додати основні MCP сервери: Playwright MCP, Automation MCP
   - Налаштувати реєстрацію і підключення MCP серверів через змінні середовища

3. **Безпечна автоматизація GUI**
   - Налаштувати безпечне середовище для Playwright в ізольованих контейнерах
   - Імплементувати macOS Automator MCP Server
   - Розробити політику авто-вибору MCP на основі метрик ефективності

4. **Налаштування безпеки системи**
   - Імплементувати Kubernetes Secrets для зберігання ключів
   - Налаштувати RBAC за принципом найменших привілеїв
   - Забезпечити аудит дій LLM3 та інших критичних компонентів

### Тести і валідація
- **Тести безпеки:**
  - Перевірка реакцій LLM3 на різні події Falco
  - Тестування ізоляції контейнерів і RBAC
  - Аудит журналів для перевірки належної реєстрації подій

- **Тести MCP Hub:**
  - Перевірка реєстрації і підключення різних MCP серверів
  - Тестування виконання команд через Playwright MCP
  - Валідація політики авто-вибору MCP на основі метрик

- **E2E тести автоматизації:**
  - Перевірка безпечного виконання Playwright-сценаріїв
  - Тестування macOS Automator інтеграції
  - Валідація метрик ефективності MCP

### Критерії приймання
- [ ] LLM3 успішно класифікує події Falco і виконує щонайменше 2 автоматизовані реакції
- [ ] MCP Hub підключає як мінімум Playwright MCP і Automation MCP
- [ ] Безпечне виконання Playwright-сценаріїв в ізольованих контейнерах працює
- [ ] Метрики ефективності MCP експортуються в Prometheus
- [ ] Система безпеки правильно обмежує доступ і фіксує події

## Фаза 4: Реалізація команд і голосових інтерфейсів

### Цілі фази
- Імплементувати Реєстр агентів і Конструктор команд
- Налаштувати TTS та STT інтеграцію
- Реалізувати Live Debate Mode

### Кроки реалізації
1. **Розробка Реєстру агентів**
   - Імплементувати CRUD операції для управління агентами
   - Розробити систему зберігання конфігурацій агентів
   - Налаштувати інтеграцію з секретами для API ключів

2. **Конструктор команд**
   - Реалізувати шаблони ролей та автопризначення
   - Імплементувати динамічне формування команд під задачі
   - Налаштувати інтеграцію з AutoGen/MetaGPT для координації агентів

3. **Інтеграція TTS (Text-to-Speech)**
   - Додати TTS MCP з підтримкою різних провайдерів
   - Налаштувати Coqui TTS як локальний фолбек
   - Реалізувати мапінг голосів до агентів LLM1/2/3

4. **Live Debate Mode**
   - Імплементувати режим живої дискусії між агентами
   - Додати STT для розпізнавання мовлення користувача
   - Розробити механізми модерації та підсумовування дискусій

### Тести і валідація
- **Тести Реєстру агентів:**
  - Перевірка CRUD операцій з агентами
  - Тестування безпечного зберігання секретів
  - Валідація інтеграції з провайдерами LLM

- **Тести Конструктора команд:**
  - Перевірка автопризначення ролей
  - Тестування динамічного формування команд
  - Валідація взаємодії між агентами в команді

- **Тести голосових інтерфейсів:**
  - Перевірка TTS з різними провайдерами і фолбеком
  - Тестування STT розпізнавання
  - E2E тести Live Debate Mode

### Критерії приймання
- [ ] Реєстр агентів дозволяє керувати агентами та їх конфігураціями
- [ ] Конструктор команд успішно формує команди під конкретні задачі
- [ ] TTS MCP працює з різними голосами для агентів з автоматичним фолбеком
- [ ] Live Debate Mode забезпечує продуктивну дискусію між агентами
- [ ] Дашборд ефективності команд показує релевантні метрики

## Фаза 5: Продакшн-готовність і фінальне тестування

### Цілі фази
- Забезпечити стабільність і надійність системи
- Впровадити комплексне E2E тестування

  - Тестування граничних випадків і відмовостійкості
  - Валідація поведінки системи під навантаженням

- **Тести фронтенду:**
  - Перевірка всіх користувацьких сценаріїв
  - Тестування відповідності UI/UX вимогам
  - Валідація інтеграції з бекендом без моків

- **Операційні тести:**
  - Перевірка процедур розгортання і оновлення
  - Тестування резервного копіювання і відновлення
  - Валідація моніторингу і оповіщень

### Критерії приймання
- [ ] Система демонструє стабільну роботу під різним навантаженням
- [ ] Всі комплексні E2E тести проходять успішно
- [ ] Базовий веб-інтерфейс працює коректно і проходить всі тести
- [ ] Документація повна і актуальна
- [ ] CI/CD конвеєр забезпечує надійне розгортання

## Загальні вимоги до тестування на всіх фазах

### CI/CD Тести
- Автоматичні CI тести для всіх компонентів при кожному коміті
- Інтеграційні тести при злитті в основні гілки
- Нічні повні E2E тести з генерацією звітів

### Unit Тести
- Покриття коду тестами не менше 80% для критичних компонентів
- Мок-тести для залежностей від зовнішніх сервісів
- Автоматизоване виявлення регресій

### Інтеграційні Тести
- Тестування взаємодії між компонентами системи
- Перевірка контрактів між сервісами
- Валідація поведінки системи при відмові окремих компонентів

### E2E Тести
- Симуляція повних користувацьких сценаріїв
- Перевірка всього ланцюга від входу до виходу
- Автоматизоване тестування UI за допомогою Playwright

### Тести Безпеки
- Статичний аналіз коду на вразливості
- Динамічне тестування на проникнення
- Перевірка правильності конфігурації RBAC і секретів

### Тести Продуктивності
- Навантажувальне тестування для визначення меж системи
- Профілювання для виявлення вузьких місць
- Моніторинг використання ресурсів під час тестів
