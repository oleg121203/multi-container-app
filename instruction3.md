# ATLAS — Об'єднана інструкція: Зведений архітектурний план та дорожня карта реалізації

## 1\. Вступ. Контекст та архітектурне бачення

### 1.1. Виконавче резюме

Цей документ слугує вичерпною дорожньою картою для проєкту ATLAS, амбітної, багатоагентної системи, спроєктованої для функціонування в динамічному, продакшен-орієнтованому середовищі. Він поєднує стратегічні архітектурні рішення, що зазвичай документуються на рівні High-Level Design (HLD), з практичними, покроковими інструкціями, що відповідають стандарту Standard Operating Procedure (SOP). Наше архітектурне бачення зосереджене на використанні моделі LLM-оркестрації, де центральний координатор (**LLM2**) керує діями спеціалізованих агентів (**LLM1**, **LLM3**, **MCP Hub**) для виконання складних, інтегрованих завдань. Ця архітектура покладається на передові технології, зокрема **Kubernetes** для оркестрації та масштабованості, **Retrieval-Augmented Generation (RAG)** для доступу до динамічних знань та багаторівневі механізми безпеки для забезпечення стійкості системи.

### 1.2. Загальне архітектурне бачення ATLAS

Архітектура ATLAS виходить за межі традиційних, монолітних систем. Вона побудована на принципах модульності та розподіленої інтелігенції. Замість єдиного універсального агента, який намагається виконати всі завдання, система делегує спеціалізовані функції експертним агентам, кожен з яких оснащений індивідуальними можливостями та інструментами. Цей підхід імітує структуру реальних організацій, де центральний координатор, або оркестратор, розподіляє завдання між спеціалізованими відділами чи командами.

Ключові компоненти архітектури ATLAS включають:

  * **Оркестратор (LLM2):** Центральний елемент, що відповідає за декомпозицію завдань, призначення інструментів та координацію потоку завдань. Він вирішує, який агент або команда агентів найкраще підходить для даного завдання, та ініціює їхню взаємодію.
  * **Спеціалізовані агенти (LLM1, LLM3):** Агенти, що виконують домен-специфічні завдання. **LLM1** спеціалізується на взаємодії з користувачем та керуванні пам'яттю через RAG, тоді як **LLM3** відповідає за аналіз та автоматичне реагування на події безпеки.
  * **Концентратор мікропроцесорних клієнтів (MCP Hub):** Шлюз, що надає агентам доступ до зовнішніх інструментів, таких як браузерна автоматизація (Playwright) та синтез мовлення (TTS), у вигляді ізольованих, контейнеризованих сервісів.
  * **Бази даних та пам'ять:** Система використовує як довготривалу (векторні бази даних **Qdrant/Milvus**), так і короткострокову (**Redis**) пам'ять для покращення продуктивності та надання актуального контексту.

## 2\. Фаза 1 — Інфраструктура (Kubernetes) та засновок проєкту

### 2.1. Перетворення `compose.yaml` → `k8s/`

Перехід від **Docker Compose** до **Kubernetes** є фундаментальним архітектурним зсувом від імперативного до декларативного управління інфраструктурою. Docker Compose чудово підходить для локальної розробки, дозволяючи інженеру вручну описувати, які контейнери запустити (`docker-compose up`). Це імперативний підхід.

Натомість, Kubernetes використовує декларативний підхід. Інженер не дає послідовних команд, а описує бажаний стан системи у YAML-маніфестах. Kubernetes-контролери постійно моніторять поточний стан кластера і працюють над тим, щоб він відповідав бажаному, забезпечуючи автоматичне управління життєвим циклом, масштабування та самовідновлення. Ця зміна необхідна для досягнення масштабованості та відмовостійкості, критичних для продакшен-середовища. Інструмент **Kompose** буде використаний для початкової конвертації (`INF-02`), але згенеровані маніфести потребуватимуть значного ручного доопрацювання для реалізації Kubernetes-специфічних концепцій.

### 2.2. Керування станом для баз даних (Qdrant/Milvus, Redis)

Ключова відмінність між *stateless* та *stateful* додатками є центральним аспектом у виборі Kubernetes-ресурсу. Stateless-додатки (веб-сервіси, агенти) не зберігають дані всередині Pods, тому для них ідеально підходить **Deployment**.

Натомість, stateful-сервіси, такі як **Qdrant, Milvus та Redis** (`INF-01`), зберігають дані, і втрата Pod може призвести до втрати інформації. Саме тому для них застосовується **StatefulSet**. `StatefulSet` надає стабільну ідентичність кожному Pod та асоціює з ним персистентний том (**PersistentVolumeClaim**). Це гарантує, що у разі перезапуску Pod він буде відновлений з тими ж мережевим ім'ям та доступом до своїх даних. Цей механізм є критично важливим для забезпечення персистентності даних у Qdrant/Milvus та для роботи Redis як кешу.

| Компонент ATLAS | Тип компонента | Рекомендований ресурс Kubernetes | Обґрунтування |
| :--- | :--- | :--- | :--- |
| Web UI / Агенти | Stateless | Deployment | Взаємозамінність Pods, легке горизонтальне масштабування. |
| Qdrant/Milvus | Stateful | StatefulSet | Стабільна ідентичність, персистентність даних, упорядковане розгортання. |
| Redis | Stateful | StatefulSet | Персистентність кешованих даних, стабільне мережеве ім'я. |

### 2.3. Оптимізація та стабільність

Для забезпечення стабільної роботи системи критично важливо правильно налаштувати механізми Kubernetes (`INF-02`).

  * **Readiness & Liveness Probes:** `Liveness Probe` перевіряє, чи працює додаток, і перезапускає його у разі збою. `Readiness Probe` перевіряє, чи готовий додаток приймати трафік, запобігаючи маршрутизації запитів до Pod, який ще ініціалізується.
  * **Керування ресурсами:** `requests` вказують на мінімальну кількість CPU/пам'яті, гарантуючи, що Pod буде запланований на відповідному вузлі. `limits` встановлюють верхню межу ресурсів, запобігаючи "шумним сусідам" та перевантаженню вузлів.
  * **Anti-affinity:** Цей механізм гарантує, що репліки критичних сервісів (наприклад, Qdrant) будуть розгорнуті на різних фізичних нодах. Це запобігає створенню єдиної точки відмови: вихід з ладу однієї ноди не призведе до одночасного збою всього сервісу.

### 2.4. Фундаментальна безпека кластера

Безпека в Kubernetes будується на принципі найменших привілеїв (`SEC-01`, `OPS-01`).

  * **NetworkPolicy:** Це ключовий інструмент для сегментації мережі. Можна створити політику "deny-all", а потім поступово додавати дозволи для критично важливих комунікацій (наприклад, дозволити LLM1 звертатися до Qdrant). Це мінімізує поверхню атаки.
  * **Role-Based Access Control (RBAC):** Дозволяє надавати специфічні дозволи на доступ до ресурсів кластера користувачам та сервісним акаунтам, знижуючи ризик несанкціонованого втручання.
  * **Kubernetes Secrets:** Замість жорсткого кодування конфіденційних даних, необхідно використовувати `Kubernetes Secrets`. Вони забезпечують шифрування даних у стані спокою (encryption at rest) і дозволяють безпечно інжектувати секрети в Pods.

### 2.5. Моніторинг та візуалізація

Ефективний моніторинг є критично важливим для підтримки здоров'я системи (`OBS-01`). **Prometheus** відповідає за збір метрик з кінцевих точок `/metrics` сервісів. **Grafana** використовується для візуалізації цих даних у зручні дашборди. Для ATLAS критично важливими є дашборди, що відображають:

  * **Метрики LLM-агентів:** Кількість запитів, затримка відповіді, кількість використаних токенів та вартість.
  * **Метрики RAG-системи:** Затримка та кількість запитів до векторної БД, частота "промахів" та "влучань" кешу.
  * **Метрики інфраструктури:** Використання ресурсів (CPU/Memory) та стану здоров'я Pods.

## 3\. Фаза 2 — Агенти та пам'ять (Core agents & RAG)

### 3.1. Архітектура RAG-системи

Система RAG (Retrieval-Augmented Generation) є основою для розширення знань LLM1, дозволяючи йому відповідати на запити, що вимагають доступу до динамічних даних (`MEM-01`). Процес RAG складається з кількох кроків:

1.  **Розбиття на фрагменти (Chunking):** Документи розбиваються на менші, осмислені фрагменти. Вибір стратегії (наприклад, `Recursive` або `Semantic chunking`) має вирішальне значення для збереження контексту.
2.  **Генерація ембедінгів (Embedding):** Кожен фрагмент тексту перетворюється на числовий вектор (ембедінг), який захоплює його семантичне значення.
3.  **Індексація:** Ембедінги індексуються у векторній базі даних (Qdrant або Milvus), що забезпечує швидкий семантичний пошук.

### 3.2. Семантичний кеш на базі Redis

Семантичний кеш на базі Redis є ключовим компонентом для оптимізації продуктивності та зменшення витрат RAG-системи. Його роль — запобігти повторним зверненням до LLM для запитів, які є семантично подібними. Коли надходить новий запит, система спочатку шукає семантично близькі запити в Redis. Якщо знайдено відповідність вище порогу схожості, повертається кешована відповідь, що значно зменшує затримку та витрати.

### 3.3. Абстракція LLM-провайдерів та `fallback`-ланцюги

Архітектура ATLAS побудована на принципі гнучкості та відмовостійкості. Замість жорсткого прив'язування до одного постачальника, система використовує проміжний шар, який дозволяє легко перемикатися між різними моделями (**OpenAI, Mistral, Gemini, Ollama**) для LLM1 та LLM3 (`CFG-01`).

Механізм **fallback** є критичною стратегією відмовостійкості. Якщо основний провайдер стає недоступним, запит автоматично перенаправляється до резервного провайдера з ланцюжка (`openai -> gemini -> ollama`). Це забезпечує безперервність обслуговування. Водночас для LLM2 існує **жорстка вимога** (`CFG-02`): він повинен використовувати локальну модель **Ollama (`gpt-oss:latest`)**. Фолбек на віддалені моделі дозволений лише за явної умови (`ATLAS_LLM2_ALLOW_FALLBACK=true`) і має суворо логуватися.

### 3.4. Матриця оцінки стратегій Chunking та Embedding

| Стратегія Chunking | Переваги | Недоліки | Рекомендоване застосування |
| :--- | :--- | :--- | :--- |
| **Fixed-size** | Простота реалізації, швидкість. | Може розрізати контекст посередині. | Загальні тексти. |
| **Recursive** | Зберігає ієрархічну структуру тексту. | Складніша реалізація. | Технічна документація, код. |
| **Semantic** | Групує речення за семантичним змістом. | Залежність від якості моделі ембедінгів. | Діалоги, дискусії. |

## 4\. Фаза 3 — Автоматизація й безпека (MCP Hub, LLM3, Falco)

### 4.1. Впровадження MCP Hub

**MCP Hub (Model Context Protocol Hub)** функціонує як централізований диспетчер, що керує ізольованими сервісами-інструментами (`MCP-01`). Він надає агентам (зокрема LLM2) доступ до зовнішніх інструментів, таких як браузерна автоматизація (**Playwright**) та синтез мовлення (**TTS**).

Критично важливим аспектом MCP Hub є **ізольоване виконання** (`GUI-01`). Playwright може взаємодіяти з потенційно шкідливими веб-сайтами. Запуск його в контейнері-«пісочниці» мінімізує ризики. Якщо Playwright відвідає шкідливий сайт, збій буде локалізований у межах ізольованого контейнера, що запобігає компрометації всього кластера.

### 4.2. Автоматичне реагування на події безпеки

Архітектура безпеки ATLAS переходить від пасивного моніторингу до активного, керованого AI-реагування (`SEC-01`). **Falco** є ядром системи виявлення загроз, спостерігаючи за системними викликами в реальному часі. Коли Falco виявляє аномальну активність (наприклад, запис у `/etc`), він генерує оповіщення, яке передається до **LLM3**.

Сценарій розгортається наступним чином:

1.  **Falco → LLM3:** Оповіщення передається до LLM3.
2.  **Аналіз та Рішення:** LLM3, спеціалізований на аналізі подій безпеки, оцінює загрозу і приймає рішення про необхідність втручання.
3.  **LLM3 → K8s API:** LLM3 формує та виконує команду для Kubernetes API. Приклади дій включають `cordon node` (ізоляція скомпрометованої ноди) або `delete pod` (негайне видалення шкідливого Pod).

### 4.3. Система TTS з `fallback`-механізмом

Для голосового інтерфейсу система використовує архітектуру з **fallback-механізмом** (`MCP-05`, `MCP-06`). Основний шлях передбачає використання **TTS MCP**, який звертається до комерційних хмарних провайдерів (Google, OpenAI). Однак для забезпечення стійкості система має локальний fallback-провайдер, такий як **Coqui TTS**. Цей локальний движок гарантує, що голосове виведення залишиться доступним навіть у разі відсутності підключення до Інтернету.

### 4.4. Матриця автоматичного реагування на події безпеки

| Тип події Falco | Конкретне правило | Аналіз LLM3 | Автоматична реакція | Обґрунтування реакції |
| :--- | :--- | :--- | :--- | :--- |
| Спроба зміни системного файлу | `Write below binary dir` | Виявлення запису у `/bin` з неавторизованого Pod. | `delete pod` | Негайне припинення потенційно шкідливого процесу. |
| Доступ до чутливого ресурсу | `Read sensitive file` | Несанкціонована спроба доступу до Kubernetes Secret. | `cordon node` | Ізоляція ноди для запобігання поширенню загрози. |
| Несанкціонований мережевий трафік | `Outbound connection to unauthorized IP` | Виявлення вихідного з'єднання з підозрілою IP. | `Block NetworkPolicy` | Негайне блокування мережевого каналу для запобігання витоку даних. |

## 5\. Фаза 4 — Команди, голос і живі режими

### 5.1. Agent Registry

**Agent Registry** — це централізована служба, яка функціонує як каталог, що містить інформацію про всі доступні агенти в системі (`TEAM-01`). Архітектура заснована на RESTful API, що дозволяє іншим компонентам динамічно знаходити агентів. Реєстр зберігає метадані та конфігурації у форматах JSON або YAML. Критично важливі секрети, такі як ключі API, зберігаються в **Kubernetes Secrets**, а не в реєстрі, що забезпечує централізоване та безпечне управління.

### 5.2. Dynamic Team Constructor

**Dynamic Team Constructor** дозволяє системі переходити від взаємодії "людина-один-AI" до "людина-команда-AI" (`TEAM-03`). Використовуючи фреймворки, такі як **AutoGen** та **MetaGPT**, LLM-оркестратор може динамічно формувати команди спеціалізованих агентів для виконання складних завдань.
Механізм роботи:

1.  Оркестратор аналізує запит користувача, щоб визначити необхідні навички.
2.  Він звертається до `Agent Registry`, щоб знайти агентів з відповідними можливостями.
3.  Оркестратор створює команду, призначаючи кожному агенту певну роль (напр., "дослідник", "кодер", "тестер") згідно з шаблонами (`TEAM-02`).
4.  Процес включає обов'язкове людське підтвердження (**human-in-the-loop**), що забезпечує контроль.

### 5.3. Голосові інтерфейси (TTS/STT)

Голосові інтерфейси є ключовим компонентом для створення природної взаємодії.

  * **Speech-to-Text (STT):** ATLAS використовує STT для перетворення мовлених запитів користувача в текст.
  * **Text-to-Speech (TTS):** TTS використовується для перетворення текстових відповідей агентів назад у мовлення. Як деталізовано у Фазі 3, система використовує **TTS MCP** з локальним **Coqui** як fallback.

### 5.4. Live Debate Mode

**Live Debate Mode** — це передова функція, що дозволяє кільком агентам брати участь у "живій дискусії", імітуючи людське спілкування.
Архітектура для цього режиму:

1.  Вхідний голосовий потік від користувача перетворюється на текст за допомогою **STT**.
2.  Оркестратор (LLM2) ініціює дискусію між агентами, обраними динамічним конструктором.
3.  Агенти обмінюються текстовими повідомленнями, а оркестратор координує їхні "виступи".
4.  Кожна відповідь агента перетворюється на мовлення за допомогою **TTS** і відтворюється для користувача.

## 6\. Фаза 5 — Підготовка до продакшену та фронтенд

### 6.1. Тестування

Перехід від прототипу до продакшен-рішення вимагає комплексного тестування.

  * **Навантажувальне тестування:** Використовується для оцінки поведінки системи під очікуваним (Load) та екстремальним (Stress) навантаженням. Інструменти, такі як Gatling, допоможуть виміряти затримку, пропускну здатність та стабільність.
  * **Безпекове тестування:** Особливу увагу слід приділити вразливостям RAG-систем, таким як **Prompt Injection** (маніпуляція запитом для виконання неавторизованих дій) та **Data Poisoning** (введення шкідливої інформації у базу знань).
  * **E2E-тестування:** Інтеграційні тести за допомогою **Playwright** (`UI-02`) перевіряють весь потік від веб-інтерфейсу до бекенду, забезпечуючи коректну роботу всієї системи.

### 6.2. Продакшен-готовність

Підготовка до продакшену включає створення надійного **CI/CD-пайплайну**, наприклад, на базі **GitHub Actions**.

  * **CI/CD Pipeline:** Типовий пайплайн складається з етапів: Source (зміна коду), Build (створення образу), Test (юніт-, інтеграційні та безпекові тести) та Deployment (розгортання на Kubernetes).
  * **Керування секретами та OIDC:** Для безпечного доступу до Kubernetes API з CI/CD рекомендується використовувати **OpenID Connect (OIDC)**. OIDC дозволяє CI/CD отримувати короткоживучі, динамічно згенеровані токени, що усуває потребу в зберіганні статичних ключів доступу.
  * **Розробка UI:** Мінімальний, але повнофункціональний веб-інтерфейс розробляється після стабілізації бекенд-контрактів. Усі дії в UI мають працювати через існуючі API без моків чи симуляцій (`UI-01`).

### 6.3. Комплексний план тестування та критерії готовності до продакшену

| Вид тестування | Мета | Інструменти | Ключові метрики | Критерії приймання |
| :--- | :--- | :--- | :--- | :--- |
| **Навантажувальне** | Оцінка пропускної здатності та стабільності. | Gatling, locust | Затримка відповіді, частота помилок (Error Rate). | Затримка \< 500мс для 95-го перцентиля, Error Rate \< 1%. |
| **Безпекове** | Виявлення вразливостей (ін'єкції, отруєння даних). | promptfoo, OWASP ZAP | Виявлені ін'єкції, витік PII. | Відсутність критичних та високих вразливостей. |
| **E2E-тестування** | Перевірка функціональності всієї системи. | Playwright, Cypress | Успішність виконання сценаріїв, коректність відповідей. | Всі критичні E2E-сценарії проходять з успіхом \> 99%. |

## 7\. Висновки, очікувані артефакти та наступні кроки

### 7.1. Підсумок ключових компонентів системи

Архітектура ATLAS створює багатогранну, адаптивну та стійку систему. Ключові рішення включають:

  * **Декларативне управління інфраструктурою:** Перехід до Kubernetes з використанням StatefulSet гарантує відмовостійкість та масштабованість.
  * **Інтелектуальне розширення знань:** Поєднання RAG з семантичним кешем на базі Redis підвищує точність та оптимізує витрати.
  * **Автоматизована безпека:** Інтеграція Falco з LLM3 та Kubernetes API переводить систему до активного, керованого AI-реагування.
  * **Мультиагентна оркестрація:** Agent Registry та Dynamic Team Constructor дозволяють створювати спеціалізовані команди агентів.
  * **Автоматизація розгортання:** CI/CD на базі GitHub Actions забезпечує відтворюваність та надійність.
