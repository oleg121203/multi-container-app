# Інструкція для GitHub Copilot на основі ATLAS.md

## 1. Контекст/Передумови
- **Опис проекту:** Створення багатоагентної системи ATLAS для автоматизації складних завдань. Система складається з трьох основних LLM-агентів з різними ролями, використовує RAG для пам'яті, MCP Hub для інструментів та AutoGen/MetaGPT для оркестрації.
- **Поточний стан:** Існує детальний архітектурний документ `ATLAS.md`, що описує всі компоненти, вимоги та план впровадження. Інфраструктура буде розгортатися в Kubernetes.
- **Релевантні документи:** [ATLAS.md](ATLAS.md)

## 2. Загальні Цілі та Завдання
- **Ціль:** Поетапно реалізувати систему ATLAS відповідно до архітектурного плану.
- **Очікувані результати:** Повністю функціональна, протестована та готова до демонстрації система, що відповідає всім критеріям приймання з `ATLAS.md`.
- **Критерії успіху:** Успішне проходження всіх E2E-тестів та приймального тестування для кожної фази.

### Фаза 1: Infrastructure MVP (Основа інфраструктури)

#### 2. Цілі та завдання
- **Ціль:** Розгорнути базову інфраструктуру в Kubernetes, готову для розміщення агентів та сервісів.
- **Очікувані результати:** Працюючий K8s кластер з розгорнутими сервісами моніторингу та персистентними сховищами для даних.

#### 3. Конкретні вимоги
- **INF-01:** Додати в `compose.yaml` сервіси для Qdrant/Milvus та Redis, забезпечивши збереження даних через volumes.
- **INF-02:** Використовуючи `kompose convert`, згенерувати початкові маніфести Kubernetes. Вручну доопрацювати їх:
  - Для Qdrant/Milvus та Redis використовувати `StatefulSet` замість `Deployment`.
  - Додати `PersistentVolumeClaim` для стабільних сховищ.
  - Прописати `readiness/liveness probes` для всіх сервісів.
  - Встановити ліміти `resources` (CPU/memory).
  - Налаштувати `anti-affinity` для розподілу критичних компонентів по різних нодах.
- **OBS-01:** Розгорнути в кластері Prometheus та Grafana для збору та візуалізації метрик.
- **SEC-01 (частково):** Створити базові `NetworkPolicy` для обмеження трафіку між сервісами. Налаштувати `Secrets` для зберігання ключів доступу.

#### 4. Тестування та CI/CD
- **Unit-тести:** На цьому етапі відсутні, оскільки робота ведеться на рівні інфраструктури.
- **Інтеграційні тести:**
  - Написати скрипт, який перевіряє доступність всіх розгорнутих сервісів (Prometheus, Grafana, бази даних) усередині кластера.
  - Перевірити, що дані, записані в Redis/Qdrant, зберігаються після перезапуску pod'а (`kill pod`).
- **E2E-тести:** Відсутні.
- **CI/CD:**
  - Створити GitHub Actions workflow, який:
    1.  Запускає лінтер для K8s маніфестів (`kubeval` або аналог).
    2.  (Опційно) Розгортає інфраструктуру в тестовий K8s-кластер (напр., `kind`).
    3.  Проганяє інтеграційні тести доступності сервісів.
    4.  Очищує тестове середовище.

### Фаза 2: Agent MVP (Ключова функціональність агентів)

#### 2. Цілі та завдання
- **Ціль:** Реалізувати ключові функції агентів LLM1 (пам'ять) та LLM2 (оркестрація), створивши основу для виконання завдань.
- **Очікувані результати:** LLM1 може запам'ятовувати контекст, а LLM2 — декомпозувати завдання та використовувати інструмент `Linear`.

#### 3. Конкретні вимоги
- **MEM-01:** Реалізувати RAG-шар для LLM1.
  - Створити логіку індексації важливих фрагментів діалогу в Qdrant/Milvus.
  - Реалізувати логіку пошуку релевантних "спогадів" та додавання їх до промпту.
- **ORC-01:** Налаштувати LLM2.
  - Підняти локальний сервер Ollama з моделлю `gpt-oss:latest`.
  - Інтегрувати фреймворк AutoGen/MetaGPT для керування агентом.
  - Створити інструмент (tool) для роботи з Linear GraphQL API (створення та оновлення завдань).
- **CFG-01:** Створити абстракцію для LLM-провайдерів (OpenAI, Mistral, Gemini, Ollama) з єдиним інтерфейсом для LLM1 та LLM3. Реалізувати логіку фолбек-ланцюга.
- **CFG-02:** Жорстко прив'язати LLM2 до локальної Ollama. Реалізувати healthcheck та механізм контрольованого фолбеку (лише якщо `ATLAS_LLM2_ALLOW_FALLBACK=true`).

#### 4. Тестування та CI/CD
- **Unit-тести:**
  - Тести для RAG-шару: індексація, пошук, форматування контексту.
  - Тести для обгортки Linear API.
  - Тести для абстракції LLM-провайдерів та логіки фолбеку.
- **Інтеграційні тести:**
  - LLM1 отримує діалог, зберігає його в Qdrant, і при наступному запиті витягує релевантний контекст.
  - LLM2 отримує завдання, і через AutoGen успішно викликає інструмент Linear, що створює реальний issue.
- **E2E-тести:**
  - Повний цикл: користувач ставить завдання LLM1 → LLM1 формалізує його → LLM2 отримує завдання, декомпозує його та створює issue в Linear. Перевірити, що issue створено з правильними даними.
- **CI/CD:**
  - Додати всі нові unit та інтеграційні тести в існуючий workflow.
  - Запускати E2E-тести (можливо, з моком Linear API або в тестовому проекті Linear) в рамках CI.

### Фаза 3: Автоматизація та Безпека

#### 2. Цілі та завдання

- **Очікувані результати:** Система реагує на загрози безпеки та може виконувати завдання в браузері в ізольованому середовищі.

#### 3. Конкретні вимоги
- **SEC-01:** Інтегрувати LLM3 з Falco.
  - Розгорнути Falco в K8s-кластері.
  - Налаштувати потік подій з Falco до LLM3.
  - Реалізувати в LLM3 логіку класифікації подій та прийняття рішень.
  - Додати мінімум дві автоматизовані реакції (напр., `cordon node`, `delete pod`) через Kubernetes API.
- **GUI-01:** Налаштувати безпечний запуск Playwright.
  - Використовувати ізольований контейнер (Apple container, Kasm, або аналог) для запуску браузера.
- **MCP-01 (частково):** Створити та розгорнути `Playwright MCP` сервер, який буде приймати завдання від LLM2 і виконувати їх у безпечному середовищі.

#### 4. Тестування та CI/CD
- **Unit-тести:**
  - Тести для логіки парсингу та класифікації подій Falco в LLM3.
- **Інтеграційні тести:**
  - Згенерувати тестову подію Falco та перевірити, що LLM3 її отримує, правильно класифікує та викликає відповідний метод для реакції (можна мокати K8s API).
  - LLM2 відправляє завдання на `Playwright MCP`, і той успішно виконує простий сценарій (відкрити сторінку, зробити скріншот).
- **E2E-тести:**
  - Згенерувати реальну подію безпеки в кластері (напр., запуск shell у контейнері) → Falco її фіксує → LLM3 реагує (напр., ізолює pod через NetworkPolicy) → дія та причина логуються.
- **Тести безпеки:**
  - Перевірити, що контейнер з Playwright дійсно ізольований і не має доступу до хостової системи чи інших pod'ів, крім дозволених.
- **CI/CD:**
  - Додати всі нові тести у воркфлоу. Запускати тести безпеки на регулярній основі.

## 4. Обмеження та Керівні принципи

### Технічні обмеження

- **Локальна Ollama для LLM2:** Строго використовувати локальну Ollama з моделлю `gpt-oss:latest`. Фолбек на віддалені провайдери лише з явним дозволом `ATLAS_LLM2_ALLOW_FALLBACK=true`.
- **Ізоляція браузера:** Playwright та інші GUI-автоматизації повинні виконуватися в ізольованих контейнерах (Apple container/Kasm) для безпеки.
- **Kubernetes-first:** Всі сервіси мають розгортатися в Kubernetes. Docker Compose використовується лише для первинної розробки.
- **Фронтенд останнім:** Веб-інтерфейс розробляється після стабілізації API. Ранні UI-прототипи мають бути повністю підключені до бекенду без моків.

### Стандарти кодування

- **Python:** Використовувати `black` для форматування, `ruff` для лінтингу, `pytest` для тестування.
- **TypeScript/JavaScript:** ESLint + Prettier для стилю, Jest/Vitest для unit-тестів.
- **Контейнери:** Multi-stage Dockerfile для оптимізації розміру образів.
- **K8s маніфести:** Використовувати `kubeval` або `kubeconform` для валідації.

### Патерни проектування

- **Dependency Injection:** Для управління залежностями між компонентами (особливо LLM-провайдери).
- **Strategy Pattern:** Для реалізації фолбек-ланцюгів LLM-провайдерів.
- **Circuit Breaker:** Для захисту від відмов зовнішніх сервісів (Ollama, Linear API).
- **Observer Pattern:** Для системи метрик та моніторингу MCP-інструментів.

### Міркування щодо продуктивності

- **RAG-пошук:** Обмежувати результати до топ-5 релевантних спогадів для контролю розміру контексту.
- **MCP-підключення:** Використовувати connection pooling та keep-alive для підключень до MCP-серверів.
- **Prometheus метрики:** Обмежувати кардинальність лейблів (максимум 10 значень на лейбл).

### Вимоги безпеки

- **Secrets:** Всі API-ключі та паролі зберігати в Kubernetes Secrets, ніколи в коді.
- **RBAC:** Мінімальні дозволи для кожного сервісу (принцип найменших привілеїв).
- **NetworkPolicy:** Обмежувати мережевий трафік між pod'ами за білим списком.
- **Аудит:** Всі дії LLM3 (реакції на події Falco) мають логуватися для розслідування.

## 6. Приклади (опційно)

### Схожі реалізації

- **AutoGen Examples:** [microsoft/autogen examples](https://github.com/microsoft/autogen/tree/main/notebook) для багатоагентних сценаріїв
- **MCP Servers:** [modelcontextprotocol/servers](https://github.com/modelcontextprotocol/servers) для референсних реалізацій
- **RAG Patterns:** LangChain або LlamaIndex інтеграції з векторними БД

### Приклади коду

```python
# Приклад фолбек-ланцюга для LLM-провайдерів
class LLMProviderChain:
    def __init__(self, providers: List[LLMProvider]):
        self.providers = providers
    
    async def generate(self, prompt: str) -> str:
        for provider in self.providers:
            try:
                return await provider.generate(prompt)
            except ProviderError as e:
                logger.warning(f"Provider {provider.name} failed: {e}")
                continue
        raise AllProvidersFailedError()
```

### Чого уникати

- **Hardcoded API keys:** Ніколи не вбудовувати ключі в код
- **Synchronous calls:** Використовувати async/await для I/O операцій
- **Large context windows:** Обмежувати розмір контексту для LLM (max 8K tokens)
- **Direct DB access:** Всі дані мають проходити через відповідні сервіси

## 8. Додаткові примітки

### Корисна інформація

- **Для налагодження:** Використовувати `kubectl logs -f` та Grafana для діагностики проблем
- **Для тестування:** Локальний K8s кластер можна розгорнути через `kind` або `minikube`
- **Для моніторингу:** Prometheus метрики доступні за адресою `/metrics` на кожному сервісі

### Питання до роз'яснення

- **Linear Project:** Який конкретний проект у Linear використовувати для тестів?
- **Apple Container:** Чи є доступ до Apple Container чи потрібно використовувати Kasm?
- **TTS Voices:** Які конкретні голоси призначити агентам LLM1/LLM2/LLM3?

### Гнучкі області vs Жорсткі вимоги

**Гнучкі області:**

- Вибір між Qdrant та Milvus для векторної БД
- UI framework для фронтенду (React vs Vue.js)
- Конкретні голоси для TTS (можна експериментувати)

**Жорсткі вимоги:**

- LLM2 повинен використовувати локальну Ollama
- Фронтенд розробляється після бекенду
- Всі тести мають бути автоматизовані в CI/CD
- Kubernetes як цільова платформа для деплою
